# Human Tokenizer 方案 v0

> 2026-07-11 · 状态：已定稿（三个待确认项均已确认，见文末）

## 一句话定位

把你的工作换算成大模型的 token 账单——一个黑色幽默向的传播型小工具，不是严肃的工作量测量仪器。

## 核心判断：定位决定一切设计

**Token 天然测不了"工作量"，只能测"文本量"。** 一句拍板的话和一万字周报，token 差一千倍，价值可能倒挂。这个缺陷藏不住，也不需要藏——它本身就是这个产品的梗：「原来我一天的产出，Opus 只要两块钱」的荒诞感，恰恰来自这种度量的不对等。

所以两条路只能选一条：

- ❌ 严肃 HR 量化工具：一定会被挑战度量有效性，两头不讨好
- ✅ 讽刺性传播产品：承认度量荒诞，把荒诞做成体验。核心指标是分享率，不是准确性

本方案按后者设计。

## 量化口径（关键设计决策）

| 口径 | 做法 | 优劣 |
|---|---|---|
| **A. 产出物口径** | 粘贴你写的文档/代码/邮件 → 直接 tokenize → 按 output token 计价 | 零后端、即贴即出结果；只测"打字量" |
| **B. 等效任务口径** | 描述你的任务 → LLM 估算"AI 做这件事要多少 input+output+迭代轮次" → 计价 | 更接近"工作量"；需要调 API（成本、key、延迟） |
| C. 时间口径 | 工时 × 模型典型吞吐 | 太抽象，放弃 |

**推荐：MVP 只做 A，V2 加 B 作为"智能估算模式"。** A 的即时性对传播型产品是致命优势——打开、粘贴、出图、分享，全程 10 秒。

## 产品形态

纯静态网页（GitHub Pages / Vercel），零后端、零服务器成本，符合开源小工具的气质。

- Tokenizer 在浏览器端跑：`gpt-tokenizer`（OpenAI 系精确）；Claude 的 tokenizer 不公开，用系数近似（约 1.1–1.25×，页面标注"估算"）
- 移动端优先：分享出去的人大多在手机上点开

## 输出设计（四层，渐进展示）

1. **API 账单**：同一份产出在各模型下的成本对比
   - 示例（2026-07 价格，output $/1M tok）：DeepSeek ≈ $1 级、Haiku 4.5 $5、Sonnet 5 $15、Opus 4.8 $25、Fable 5 $50
   - 文案方向：「你今天的产出，Opus 4.8 只收 ¥2.3」
2. **订阅额度视角**：折算成 Claude Pro/Max、ChatGPT Plus 的额度消耗百分比
   - 官方不公开额度，用社区估算值 + 明确标注"娱乐向估算"
   - 文案方向：「你这周的工作 = Claude Pro 月度额度的 3%」
3. **单价对决（爆点）**：输入月薪 → 算出你的每 token 单价 vs 模型单价
   - 文案方向：「你比 Opus 4.8 贵 1800 倍，但你不能并发，还要交社保」
4. **分享卡片**：一键生成可保存的图片卡片——这是传播的全部关键，梗密度投入优先级最高

## 数据维护

- 模型价格：单独 `pricing.json` 配置文件，可参考 litellm 维护的 model_prices 数据源，社区可 PR 更新
- 订阅额度：手动维护的估算表，每条带来源链接和"估算"标注

## 风险与克制

1. **一波流产品**：生命周期可能就几周，接受它。投入重心放在分享卡片和文案，不做用户系统、不做历史记录、不过度工程化
2. **额度数据会被挑刺**：全部标注估算 + 来源，评论区吵起来反而是传播
3. **Claude token 数是近似值**：标注即可，误差 20% 不影响幽默效果
