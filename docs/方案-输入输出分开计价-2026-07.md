# 方案：输入 / 输出分开计价（吞吐比）

日期：2026-07-16　状态：已实施（2026-07-16，本地已验证，未部署）

## 第一性原理：问题到底是什么

现在的口径是「产出物口径」：用户粘贴的文本 = 交付物 = 模型的 **output**，按输出价计价。这个隐喻本身没错——**当前实现不是 bug，是维度缺失**。

真实的 LLM 账单永远是两行：input + output。对应到人：

| API 账单 | 人的工作 |
|---|---|
| input tokens | 你为了干活吃进去的东西：需求文档、会议纪要、群消息、被 @ 的长文 |
| output tokens | 你实际交付的东西：周报、方案、代码、PPT |

区分输入输出的真正价值不是「计价更准」（本来就是娱乐向估算），而是解锁一个**新的扎心数字：输入产出比**。「你这周吃进 8 万 token 的会议纪要，产出 600 token 的周报，蒸馏效率 0.75%」——这比单纯的账单金额更有传播力，且天然讽刺「用 token 度量人」这件事。核心指标是分享率，这个维度直接服务于它。

## 隐喻映射（已定即为规则）

- **产出区（现有输入框）** = output，按输出价计价。语义、位置、默认体验全部不变。
- **投喂区（新增，默认折叠）** = input，按输入价计价。文案方向：「你为了产出这坨，吃进了什么？」
- 两者都填 → 账单像真发票一样分两行 + 合计，并出**吞吐比**。
- 只填产出 → 行为与现在完全一致。**不填投喂区的用户体验零变化**——这是不伤分享漏斗的底线。

### 否掉的备选

- ❌ 「模式切换」（让用户声明粘的是输入型还是产出型工作）：要求用户先思考分类，违反不言自明原则。
- ❌ 每个模型展示输入/输出两个单价表：工具计价的是具体的一坨文本，不是价目表，展示单价不产生扎心数字。
- ❌ prompt caching 折扣价、阶梯计价：精度收益撑不起复杂度。转化为吐槽文案：「人类没有 prompt cache，第二遍读需求照样全价计费」。

## 各模块影响

### 数据层（data/）

- `pricing.json`：**零改动**。所有模型的 `input_usd_per_mtok` / `input_cny_per_mtok` 已齐备且带来源。
- `i18n.json`：新增 keys（投喂区标题/占位、账单 input/output 行、吞吐比文案、caching 吐槽）。中英分开创作，不逐字互译。
- `plans.json`：零改动，但额度占比改按 input+output 总 token 计算——真实订阅额度就是输入也烧额度，开会也在烧命，这本身是梗。

### 逻辑层（js/pricing.js）

- 新增 `inputUsdPerMtok(m)`（与现有 `outputUsdPerMtok` 对称）。
- `apiBill(outputTokens)` → `apiBill(outputTokens, inputTokens = 0)`：返回每模型 `{ inUsd, outUsd, totalUsd }`，排序改按 total。
- `planUsage` 改用总 token。
- `humanRate` 不变：人的成本 = 月薪 × 耗时，本来就读写一体计费。对决语义升级为「你的总成本 vs 模型跑同样 input+output 的总账单」。
- `vendorTopModel` 维持按输出价选旗舰，不动。

### 界面层（index.html / app.js / css）

- step ①「投喂」区下方加折叠入口（一行文字按钮，如「+ 顺便算算你吃进了什么」），展开后是第二个 textarea + 复用现有文件上传。渐进式展示：默认视野里只有一个框。
- step ②「账单」：有 input 时每模型仍单行、金额取合计，表头注释说明双轨口径（23 个模型 × 两行会把列表拉到两倍长，明细密度不值得；in/out 拆分放分享卡片）；无 input 时维持现状。
- 结果区新增吞吐比数字（output/input 百分比），作为第一屏扎心数字的候补——只在填了投喂区时出现。

### 分享卡片（js/card.js）

- 有 input 时卡片加「吞吐比」和 input/output 两行小账单。这是本次迭代的传播核心，梗密度优先投入在这里。

## 文案方向（示意，中文侧）

- 吞吐比命名候选：「蒸馏效率」「吞吐比」「消化率」
- 「吃进 82,400 tok / 产出 617 tok——按这个蒸馏效率，你早被降级到 light tier 了」
- 免责标注沿用：估算值必须带「娱乐向估算」

## 风险

1. **多一个框伤首次转化**：防线 = 默认折叠 + 不填零影响。上线后看分享卡片里带吞吐比的占比，低就说明折叠入口文案要改，而不是功能没价值。
2. **吞吐比可能反向冒犯**（讽刺个人而非讽刺度量本身）：文案矛头必须始终指向「用 token 度量人」的荒谬，自嘲不嘲人。
